# TDD Workflow 技能

受 ECC 项目启发，专为 Codex 优化的测试驱动开发工作流。

## 描述

TDD Workflow 技能指导 Codex 以严格的红-绿-重构循环进行开发，确保代码质量和测试覆盖率。

## 使用场景

- 新增功能开发
- 重构现有代码
- 修复 bug（先写复现测试）
- 代码审查（验证测试完整性）

## 工作流程

### Phase 1: Red — 编写失败的测试

```
1. 理解需求，但不写实现代码
2. 编写描述性的测试用例
3. 运行测试，确认测试失败（符合预期）
4. 检查失败信息是否清晰
```

### Phase 2: Green — 使测试通过

```
1. 编写最简单的实现代码使测试通过
2. 不需要追求完美，先让测试变绿
3. 可以硬编码返回值（暂时）
4. 运行测试，确认测试通过
```

### Phase 3: Refactor — 重构

```
1. 在测试通过的前提下改进代码
2. 消除重复
3. 提高可读性
4. 运行测试，确认仍然通过
5. 检查代码覆盖率
```

## 示例

### 任务示例

```
使用 TDD 实现一个用户注册功能，包含：
- 邮箱验证
- 密码强度检查
- 重复邮箱检测
```

### Codex 指令模板

```
请使用 TDD 工作流帮我实现以下功能：

需求：[具体需求描述]

请遵循以下步骤：
1. 先编写测试用例，覆盖正常路径和边界情况
2. 运行测试确认失败
3. 编写最简实现使测试通过
4. 重构代码，确保质量和可读性
5. 验证测试覆盖率达到 80%+

技术栈：[TypeScript/Python/Go 等]
```

## 检查清单

- [ ] 测试在实现之前编写
- [ ] 每个测试都有明确的断言
- [ ] 测试覆盖正常路径和异常路径
- [ ] 实现代码刚好使测试通过（不过度设计）
- [ ] 重构后代码更清晰
- [ ] 测试覆盖率 >= 80%
- [ ] 所有测试通过
